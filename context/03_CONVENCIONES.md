# 03. CONVENCIONES Y REGLAS DE CÓDIGO

## 1. Diseño y Estilos (UI)
- **USAR EXCLUSIVAMENTE SLDS:** Todo nuevo diseño debe utilizar clases nativas del Salesforce Lightning Design System (`slds-box`, `slds-grid`, `slds-button`, `slds-input`).
- **PROHIBIDO TAILWIND:** TailwindCSS no está instalado ni procesándose en el pipeline de Vite. Utilizar clases como `flex`, `bg-blue-500`, `p-4` no tendrá ningún efecto y romperá el diseño.
- Si SLDS no cubre una necesidad específica de diseño (e.g., alturas o posicionamientos exactos), utilizar CSS Modules (`.module.css`) nativos para componentes aislados.

## 2. Acceso a Datos y Base de Datos
- **Manejo de Promesas:** Utilizar siempre `async/await` al interactuar con Supabase.
- **Consultas (Queries):** Emplear el cliente estándar: `const { data, error } = await supabase.from('tabla').select('*')`.
- **Multi-Tenancy:** Cualquier consulta que haga un gerente (`manager`) o agente (`agent`) DEBE filtrar obligatoriamente por el `tenant_id` actual para evitar fugas de información inter-inmobiliarias.

## 3. Estado de Componentes
- Favorecer la composición funcional y React Hooks (`useState`, `useEffect`).
- Componentes pesados (como dashboards con múltiples gráficos analíticos) deben abstraer su lógica de fetching de datos en Custom Hooks (ej. `useAdminMetrics.ts`).

## 4. Reglas Estrictas de Renderizado UI (Añadidas el 25-06-2026)
- **Top Bar (Page Header):** Está prohibido usar el fondo gris por defecto de `slds-page-header`. Debe forzarse el fondo a blanco o transparente para mantener la estética limpia.
- **Selects en Tablas:** Los `<select>` nativos dentro de las tablas SLDS deben llevar explícitamente `appearance: 'none'` (o equivalente en CSS) para evitar que se renderice doble flecha (la del navegador y la del SLDS).
- **Espacio Celeste Superior (Brand Band / Background):** El espacio celeste en la parte superior de la aplicación debe tener una altura fija y consistente en absolutamente todas las páginas del proyecto, sin importar el contenido de la página.

## 5. Reglas Estrictas de Arquitectura CSS e Íconos
Para evitar la desalineación entre el marco SLDS estático y nuestros componentes React dinámicos, debes acatar estas reglas absolutas de diseño:
1. **Íconos de Lucide-React (`<Search />`, `<Plus />`, etc):** No agregues `style={{color: '...'}}` a cada ícono individual. El archivo `index.css` global ya tiene reglas maestras que obligan a cualquier `<svg>` de Lucide a tener `fill: none !important` y heredar automáticamente su color con `stroke: currentColor !important`. Solo necesitas envolver los íconos de Lucide con las clases semánticas correctas de SLDS (ej. `slds-icon`). **NUNCA** verás un ícono "manchado de negro" si sigues esto.
2. **Modales y la 'X' de cerrar:** En nuestro diseño moderno, la "X" del botón de cerrar va **dentro** de la caja blanca de cabecera del modal (`slds-modal__header`). Por lo tanto, **NUNCA utilices la clase `slds-button_icon-inverse`** en la X de un modal. Esa clase generará un recuadro blanco roto sobre fondo blanco.
3. **Fondo Global (Brand Band):** El diseño actual ya no utiliza un "corte" de color superior (Brand Band falso). El contenedor principal (`.contentScroll`) tiene un color de fondo corporativo continuo y sutil (`#eaf5fe`). No modifiques `CorporateLayout.module.css` para inyectar pseudo-elementos (`::before`) que corten el fondo.
4. **Selects Nativos y Flechas Dobles:** Está **ESTRICTAMENTE PROHIBIDO** usar estilos en línea como `style={{ appearance: 'none' }}` en los componentes de React, ya que pueden generar conflictos con navegadores en Windows y volver a mostrar la doble flecha. La única forma válida de ocultar las flechas nativas es usar la clase `className="slds-select"`, la cual ya está configurada globalmente en `index.css` con `!important`, `text-indent: 0.01px` y `text-overflow: ''` (el hack nuclear necesario para Firefox/Edge).
5. **Evitar "Double Padding" en Páginas:** El layout maestro (`CorporateLayout.module.css` en la clase `.contentScroll`) ya provee un `padding` corporativo global (16px arriba/abajo, 24px lados) para todas las rutas. **ESTÁ PROHIBIDO** envolver el nodo raíz (`div`) de una nueva página en clases de padding adicionales (ej. `className="slds-p-around_medium"`). Esto causa "doble padding", encogiendo el contenido y rompiendo el uso eficiente del lienzo (Data Density). Los componentes de nivel de página deben aprovechar el 100% del ancho disponible sin márgenes extra.
6. **Diseño 100% Fluido (Fluid Layout):** El sistema está diseñado para consumir pantallas Ultrawide and 4K (32" o más). En `CorporateLayout.module.css`, la clase `.contentWrapper` tiene `max-width: 100%`. Está estrictamente prohibido imponer anchos máximos estáticos (ej. `max-width: 1600px` o `container-xl`) que "enjaulen" el contenido en el centro de la pantalla, ya que esto interrumpe la densidad de datos de las tablas operativas.
7. **Arquitectura Cero Custom CSS (Pure SLDS):** Queda **ESTRICTAMENTE PROHIBIDO** crear archivos `*.module.css` para estilizar tarjetas, tablas o contenedores genéricos (como se hizo antiguamente en el módulo de Reportes). Todo el diseño visual, espaciado (`slds-p-*`, `slds-m-*`), cuadrículas (`slds-grid`), componentes (`slds-card`, `slds-panel`) e íconos debe construirse utilizando única y exclusivamente las clases de utilidad del Salesforce Lightning Design System (SLDS). El uso de estilos en línea o módulos CSS solo se permite para casos de extrema excepcionalidad (ej. animaciones hiper-específicas). La arquitectura de referencia oficial del proyecto es el código de `CompaniesDashboard` (Inmobiliaria).
8. **Fondos de Tablas Modernas (No Zebra Striping):** Está estrictamente prohibido el uso de la clase `slds-table_striped` (fondos grises intercalados) en las tablas de datos, ya que incrementa la carga cognitiva y se considera un patrón obsoleto. Las tablas deben tener un **fondo 100% blanco** utilizando únicamente las clases `slds-table slds-table_cell-buffer slds-table_bordered` y agregando obligatoriamente `slds-table_hover` para la retroalimentación interactiva. La separación visual de datos se logrará exclusivamente mediante bordes sutiles y paddings, nunca mediante sombreados de filas estáticos.
9. **Íconos Estándar (Interoperabilidad Lucide-SLDS):** Al utilizar íconos de `lucide-react` dentro de contenedores estándar de Salesforce que tienen fondos de color (ej. `className="slds-icon_container slds-icon-standard-account"`), es **obligatorio** inyectar explícitamente `style={{ color: 'white' }}` en el contenedor `<span>`. Dado que los íconos Lucide usan `stroke: currentColor`, si no se fuerza el color a blanco, el ícono heredará el color oscuro del texto circundante y se dibujará negro sobre el fondo de color, lo cual es un bug visual.
10. **Arquitectura Multi-Tenant estricta (Owner vs Tenant Admin):** Queda prohibido mezclar la configuración operativa del SaaS con la configuración de negocio de los clientes. El rol `owner` (dueño de la plataforma) solo debe tener acceso a **Operaciones SaaS** (Facturación, Planes, Auditoría Global, Plantillas). La configuración técnica del negocio (Roles, SLA, Automatizaciones, Modelos de Datos Custom) pertenece exclusivamente al rol `manager` (Tenant Admin) bajo el módulo **Configuración de Inmobiliaria**. El `owner` nunca debe configurar reglas de negocio globales que rompan la autonomía de los tenants.
11. **Uso de Pestañas (SLDS Tabs):** Al construir navegación por pestañas (`slds-tabs_default`), está prohibido inyectar íconos (`lucide-react`) dentro de las etiquetas `<a>` de navegación (`slds-tabs_default__link`). SLDS no maneja bien el alineamiento flex de estos íconos, rompiendo la UI (stack vertical y superposición). Usa solo texto. Además, el contenedor de las pestañas debe ser siempre un `slds-card` estándar con fondo blanco; nunca forzar `backgroundColor: 'transparent'` ya que los tabs requieren un fondo sólido para delimitar correctamente el área activa.
12. **Estrategia de Integración (Webhooks vs Middleware):** A nivel arquitectónico, el sistema provee integraciones "Plug & Play" mediante **Webhooks Nativos** (URLs únicas por Inmobiliaria) para capturar leads de Facebook Ads y mensajes de WhatsApp. Queda prohibido obligar al usuario (Tenant) a utilizar plataformas middleware de pago como Zapier o Make para operaciones core de captura de leads. El CRM actúa como su propio motor de ingestión y asignación (Round-Robin).
13. **SLDS Table Headers:** Evita usar `slds-table_header-fixed_container` y `slds-table_header-fixed` a menos que estés dispuesto a inyectar padding manual estricto. Si los usas "out of the box", el `padding-top` faltante causará que el color de fondo gris se escape por encima de la cabecera. Para tablas simples con scroll, prefiere usar solo `<table className="slds-table">` envuelto en un `slds-scrollable_y`.
